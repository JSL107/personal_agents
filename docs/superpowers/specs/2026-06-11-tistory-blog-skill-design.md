# Tistory 블로그 자동 초안 스킬 — 설계 (Hermes `tistory-blog`)

- 작성일: 2026-06-11
- 대상 시스템: **Hermes Agent** (`~/.hermes/`) — 이 레포(이대리/personal_agents)가 아님. 본 문서는 이 레포의 superpowers 기획 문서 관례(§docs/superpowers)에 따라 여기 보관하되, 구현물은 `~/.hermes/skills/tistory-blog/`에 위치한다.
- 블로그: [이런 저런 이야기](https://js-lee.tistory.com) (JS.Lee)
- 상태: 설계 확정 (구현 계획 대기)

---

## 1. 배경 / 목적

사용자가 자연어로 "이것에 대해 블로그 써줘"라고 요청하면, 봇이 **알아서 자료를 조회/정리해 템플릿에 맞는 글 초안**을 만들어 준다. 주제는 주로 **IT 기술**과 **본인이 한 작업 회고**. 발행 대상은 본인 블로그 `js-lee.tistory.com`.

초안은 Notion에 정리되고, Slack DM으로 Notion 링크가 전달된다. 사용자는 Notion에서 검토·수정 후 Tistory 에디터에 붙여넣어 **직접 발행**한다.

## 2. 핵심 결정 (확정)

| 항목 | 결정 | 이유 |
|---|---|---|
| 발행 방식 | **초안 생성 + 수동 발행** | Tistory Open API가 2024-02 종료되어 자동 발행 불가. 견고·안전·검토 가능. |
| 호스트 | **Hermes 스킬** | 웹검색·git/PR(terminal)·메모 확장이 모두 Hermes 네이티브. 이대리(NestJS)는 결정적 단발 CLI 구조라 agentic 리서치와 안 맞음. |
| 콘텐츠 소스 | **웹 / git·PR / 메모** 3모드 | 사용자가 셋 다 사용. |
| 정리처 | **Notion 전용 DB '블로그 초안'** | 검토·수정·상태관리 용이. 개인 용도(회사 거버넌스 무관). |
| 전달 | **Slack DM(idaeri 봇) + Notion 링크** | 아침신문과 동일 발신 경로 재사용. |
| 트리거 | **터미널 `hermes "<요청>"`** | Hermes는 Slack 수신 불가(아래 §11). |
| 말투 | **[#4 글](https://js-lee.tistory.com/4) 보이스** | §7 참조. |

## 3. 비목표 (Out of Scope, YAGNI)

- 자동 발행(브라우저 자동화) — Tistory/카카오 봇감지·DOM 취약·ToS 회색지대로 제외.
- 이미지/썸네일 자동 생성 — 후속(Hermes `image_gen` 보유).
- Tistory 외 플랫폼.
- **Slack에서 *요청* 받기** — Hermes 전용 Slack 앱(별도 토큰/소켓) 필요. 후속 검토.

## 4. 아키텍처 / 흐름

```
hermes "<요청>"            (터미널 일회성; `hermes` 단독은 대화형 TUI)
  │
  ├─[tistory-blog 스킬]
  │   1) 소스 모드 판별 (web / pr / note)
  │   2) 리서치:  web_search(DDGS)  |  git·gh(terminal)  |  메모 파싱
  │   3) 초안 작성: 글유형 템플릿 + #4 말투 (markdown)
  │   4) 메타 생성: 제목 후보 2~3, 카테고리, 태그 5~7, 출처유형
  │
  ├─[notion 스킬]  '블로그 초안' DB에 페이지 생성 (속성 + 본문 블록)
  │
  └─[slack 발신]   idaeri 봇 DM → 제목 + Notion 링크 + 1줄 요약 + 카테고리/태그
                    ↑ 이것이 사용자가 받는 "답장"

사용자: Notion 검토·수정 → Tistory 마크다운 에디터 붙여넣기 → 카테고리/썸네일/공개설정 → 발행
```

구현물은 마크다운 지시 스킬(`SKILL.md` + `references/`, `templates/`)이며, Hermes 에이전트 루프가 이를 따라 기존 도구(web_search, terminal, notion 스킬의 `ntn`/curl, slack 발신)를 오케스트레이션한다.

## 5. 트리거 & UX

- 일회성: `hermes "React 서버 컴포넌트 CS 글 써줘"`
- 회고: `hermes "vacation 기능 PR 회고 블로그 써줘, 레포: ~/Desktop/backend/기타/personal_agents"`
- 메모: `hermes "이 메모로 글 확장해줘: <메모/개요/링크>"`
- 대화형: `hermes` (TUI)에서 동일 요청. 대화형에서는 모드/레포가 모호하면 **되물을 수 있다**. 일회성에서는 되묻지 못하므로 **모드를 추론하고 가정을 Notion 페이지 상단 콜아웃에 명시**한다.

## 6. 소스 모드

1. **git/PR 회고 (`pr`)** — 가장 강력. terminal로 `git -C <repo> log/show`, `gh pr view/diff`를 읽어 실제 변경 기반 회고. 요청에 레포 경로 포함. 미지정 시 `hermes` 실행 디렉토리를 레포로 간주하고, 그곳이 git 레포가 아니면 — 일회성은 명확한 실패 메시지로 중단(추측 금지), 대화형은 경로를 되묻는다. PR 번호/브랜치 지정 가능.
2. **메모 확장 (`note`)** — 강력. 요청에 포함된 메모/개요/불릿을 완성된 글로 확장. 링크가 있으면 검색 시드로만 사용(본문 추출 불가, §10).
3. **웹 리서치 (`web`)** — 제한적. DDGS 검색 **스니펫 + 모델 지식**으로 작성하고 **출처 URL을 본문 말미에 명시**. 본문 추출 없음. 검색은 **순차 호출**(동시 호출 시 rate-limit, 아침신문 교훈).

## 7. 글 템플릿 + 말투 규칙

### 말투 (모든 유형 공통 기본 보이스 — [#4](https://js-lee.tistory.com/4) 기준)
- 종결어미: **`~했다 / ~다` 과거형 평서체** (존댓말 `~합니다/~해요` 금지). 문어체 기본 + 구어 표현 혼합.
- 1인칭 **"나"** (저/필자 금지). 예: "나는 ~하게 됐다", "내게 낯선 생태계였다".
- 톤: **"선배의 경험담"** — 친근하되 성찰적. 긴 설명 뒤 짧은 임팩트 문장.
- 말버릇: "막상 ~보니", "결과적으로", "사실", "~느꼈다 / 배웠다", "가장 크게 느낀 건 ~라는 점이었다".
- 구조 기호: 소제목 `###`, 번호목록 `1) 2) 3)`, 섹션 구분 `---`, 핵심 **굵게**, 인용 `>`.
- 문단 3~5줄. **이모지 거의 미사용**(굵은 글씨·가로선으로 강조). 마무리는 통찰 한 문장.

### 글유형 템플릿 (요청에 따라 선택)
- **개발 회고** (pr): 배경·문제 → 접근/구현 → 막힌 점·해결 → 회고·배운 점
- **기술·CS** (web): 개념 → 동작 원리 → 예시/코드 → 정리 + 출처 링크
- **알고리즘** (코딩테스트): 문제 → 접근 → 코드 → 복잡도·회고

길이: 500~1500자(블로그 기존 글 호흡). 코드블록·소제목 적극 사용.

## 8. Notion '블로그 초안' DB

### 스키마
| 속성 | 타입 | 값 |
|---|---|---|
| 제목 | title | 글 제목(대표 1개) |
| 카테고리 | select | 코딩테스트 준비 / CS / 개발 회고 / Nest & Spring / 아무거나 |
| 태그 | multi-select | 5~7개 |
| 상태 | select | 초안 / 검토 / 발행 |
| 출처유형 | select | 웹 / PR / 메모 |
| 생성일 | date | 생성 시각 |

### 페이지 본문
- 상단 콜아웃: 제목 후보 2~3개 + (일회성 추론 시) 가정/모드 명시 + (web) 출처 링크 요약.
- 본문: 초안 markdown → Notion 블록(헤딩/코드블록/번호목록/인용). `ntn` 설치 시 markdown import, 미설치 시 curl로 블록 구성.
- 상태 기본값 = **초안**.

## 9. 셋업 전제 (사용자 준비 항목)

1. **Notion integration 토큰**: notion.so/my-integrations 에서 생성 → `~/.hermes/.env` 에 `NOTION_API_KEY=ntn_...`.
2. **'블로그 초안' DB 생성** (위 스키마) + 해당 DB를 integration에 **공유**(`...` → Connect to) → **database_id 확보**.
3. DB id 전달 방식: `~/.hermes/.env` 에 `TISTORY_BLOG_DB_ID=<id>` (스킬이 읽음).
4. (선택) `ntn` CLI 설치(`curl -fsSL https://ntn.dev | bash`) — 없으면 curl 폴백.
5. Slack 발신: idaeri 봇 토큰 재사용(이미 설정). 대상 DM 채널은 아침신문과 동일.
6. `gh` CLI 인증(PR 모드) — 이미 사용 중.

## 10. 알려진 제약 / 리스크

- **웹 리서치 깊이**: DDGS는 search-only(본문 추출 X) + 가끔 rate-limit. 깊은 글은 메모 모드로 자료 보강 권장. 후속 견고화: Brave API(`BRAVE_SEARCH_API_KEY`, 월 2000건 무료).
- **일회성 트리거의 클러리피케이션 부재**: 모드/레포 모호 시 추론 + 가정 명시로 대응.
- **Notion markdown→블록 충실도**: 복잡한 마크다운(중첩·표)은 변환 손실 가능 → 코드블록/헤딩/목록 위주로 안전하게.
- **Tistory 붙여넣기**: Tistory 마크다운 모드 기준. HTML 모드 사용 시 변환 필요할 수 있음(기본은 마크다운 모드 가정).
- **Hermes Slack 수신 불가**(§11).

## 11. Hermes Slack 제약 (중요 전제)

Hermes는 **발신 전용**이다. 아침신문 설정 시 `SLACK_APP_TOKEN`(Socket Mode 수신)을 **일부러 넣지 않았다** — 이대리 NestJS가 같은 앱 토큰으로 이미 Socket Mode 수신 중이라, Hermes가 두 번째 소켓을 열면 Slack이 이벤트를 두 연결에 나눠 보내 **이대리가 망가지기** 때문. 따라서 블로그 요청은 **터미널**에서 하고, 결과만 Slack DM으로 푸시한다.

## 12. 검증 기준 (Acceptance)

1. `hermes "<주제> 블로그 써줘"` 실행 시:
   - '블로그 초안' DB에 페이지가 생성되고 속성(제목/카테고리/태그/상태=초안/출처유형/생성일)이 채워진다.
   - 본문이 §7 말투(`~했다` 평서체, "나" 1인칭, 이모지 미사용)와 해당 글유형 템플릿을 따른다.
   - Slack DM에 제목 + Notion 링크 + 1줄 요약 + 카테고리/태그가 도착한다.
2. PR 모드: 지정 레포의 실제 PR/커밋 내용이 회고에 반영된다.
3. 웹 모드: 출처 URL이 본문 말미에 포함된다.
4. 초안 markdown이 Tistory 마크다운 에디터에 붙여넣기 가능한 형태다.
