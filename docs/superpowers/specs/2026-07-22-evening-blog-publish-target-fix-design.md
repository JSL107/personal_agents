# 저녁 회고 블로그 발행 — 적재 목적지 교정

**날짜**: 2026-07-22 · **상태**: 설계 승인 대기 · **선행 조사**: 운영성 감사 후속 G7 판단

## 배경

"저녁 회고 블로그 발행 버튼을 눌러도 발행이 안 된다"는 증상에서 출발했다. 조사 결과 **발행 자체는 정상 동작하고 있었고, 글이 의도와 다른 위치에 쌓이고 있었다.**

### 실제로 일어나던 일

`EveningBlogPublishApplier`는 `EVENING_RETRO_BLOG_NOTION_PAGE_ID`가 가리키는 페이지 밑에 자식 페이지를 만든다(`evening-blog-publish.applier.ts:79-91`). 그런데 이 값이 경력 로그 페이지와 동일하게 설정돼 있었다.

```
EVENING_RETRO_BLOG_NOTION_PAGE_ID = 34b69cbba394807588c4e2e5b95f4be9   → "학교종이" (경력 로그)
CAREER_LOG_NOTION_PAGE_ID         = 34b69cbba394807588c4e2e5b95f4be9   → 동일
```

의도한 적재 위치는 "블로그 초안" 데이터베이스(`37c69cbba394806a983fc42d03a12f19`)다. 자연어 경로(`@이대리 ... 블로그 써줘` → Hermes)는 이미 이 DB에 쓰고 있어, **두 경로가 서로 다른 곳에 결과를 남기고 있었다.**

### 근거

승인 기록과 Notion 실물이 일치한다. 즉 applier는 실패한 적이 없다.

| 승인(APPLIED) 시각 | Notion 자식 페이지 last_edited |
|---|---|
| 2026-07-04 10:04:47 | 2026-07-04T10:04 |
| 2026-07-06 10:05:03 | 2026-07-06T10:05 |

"블로그 초안" DB에는 자연어 경로가 만든 글 3건이 들어 있다(루프 엔지니어링 2026-07-07, nginx·gateway 2026-06-25, HTTP 캐시 2026-06-11). 저녁 회고 글은 이 목록에 없다.

## 목표

저녁 회고 발행 결과를 "블로그 초안" DB에 행으로 적재해, 자연어 경로와 같은 곳에서 관리되게 한다.

**완료 기준**: 승인 버튼을 누르면 "블로그 초안" DB에 새 행이 생기고, 제목·태그·출처유형·카테고리·상태가 채워지며, 본문이 그 행 안에 들어간다.

## 설계

### 적재 방식

DB 행 생성은 `NotionClientPort`에 이미 있는 `findOrCreateDailyPage`를 쓴다. 이 메서드는 대상 DB의 title 속성명을 조회해 같은 제목의 행이 있으면 재사용하고 없으면 생성하므로(`notion-api.client.ts:253-291`) 재승인 시 중복 행이 생기지 않는다. 새 포트 메서드는 필요 없다.

```
현재:  findOrCreateChildPage({ parentPageId })   → 페이지 밑 자식 페이지
변경:  findOrCreateDailyPage({ databaseId })     → DB 행
       updatePageProperties({ pageId, ... })     → 속성 채우기
       appendBlocks({ pageId, blocks })          → 본문 (변경 없음)
```

### 채울 속성

대상 DB의 기존 선택지에 맞춘 값만 사용한다.

| 속성 | 타입 | 넣을 값 | 비고 |
|---|---|---|---|
| 제목 | title | `payload.topPick.title` | `findOrCreateDailyPage`가 설정 |
| 태그 | multi_select | `payload.topPick.keywords` | 신규 값은 Notion이 옵션에 자동 추가 |
| 출처유형 | select | `PR` | 저녁 회고는 그날 머지된 PR 기반 |
| 카테고리 | select | `개발 회고` | 기존 옵션과 일치 |
| 상태 | select | `초안` | 기존 행 3건과 동일 |

속성명과 값은 도메인 상수로 고정한다. 자연어 경로는 속성명을 env로 덮어쓸 수 있게 돼 있지만(`BLOG_NOTION_PROP_*`), 저녁 회고용으로 같은 env를 네 개 더 만드는 것은 이득 없이 설정만 늘린다. DB 속성명이 바뀌면 그때 대응한다.

### 환경 변수

`EVENING_RETRO_BLOG_NOTION_DATABASE_ID`를 새로 도입한다. 기존 `EVENING_RETRO_BLOG_NOTION_PAGE_ID`는 의미가 달라졌으므로(페이지 → DB) 이름을 바꿔 혼동을 없앤다. 미설정 시 지금과 동일하게 명시적 오류로 중단한다.

변경 대상 4곳: `.env.example`, `.env`, `src/config/app.config.ts`, README 표.

### 속성 설정 실패 처리

**실패를 삼키지 않고 사용자에게 알린다.** 자연어 경로가 속성 설정 실패를 `logger.warn`으로만 남기고 `published: false`를 조용히 반환해 온 탓에, 실행 4건이 전부 실패하고 있다는 사실이 이번 조사 전까지 드러나지 않았다(`generate-blog-draft.usecase.ts:123-130`). 같은 함정을 반복하지 않는다.

본문 적재까지 끝난 뒤 속성 설정만 실패한 경우에는 글이 유실되지 않았음을 함께 안내한다.

## 동작 변화 / 리스크

- **적재 위치가 바뀐다.** 배포 후 저녁 회고 글은 경력 로그 페이지가 아니라 "블로그 초안" DB에 쌓인다. 기존에 경력 로그에 쌓인 2건은 그대로 두며 자동 이관하지 않는다.
- **환경 변수 이름이 바뀐다.** `.env` 갱신 없이 배포하면 발행이 명시적 오류로 중단된다. 조용한 실패보다 낫다고 판단했다.
- `findOrCreateDailyPage`의 제목 기준 재사용 때문에, 같은 제목의 글을 다시 승인하면 새 행 대신 기존 행에 본문이 덧붙는다. 저녁 회고는 하루 한 건이고 제목에 그날 주제가 들어가므로 실사용에서 충돌 가능성은 낮다.

## 검증

- 단위 테스트: `evening-blog-publish.applier.spec.ts` — DB 행 생성 호출 인자, 속성 payload 구성, 본문 append 순서, DB ID 미설정 시 오류, 속성 설정 실패 시 안내 문구.
- 게이트: `pnpm lint:check` / `pnpm test`(전체) / `pnpm build` 모두 통과.
- 실증: 배포 후 저녁 승인 카드로 실제 발행 1회. "블로그 초안" DB에 행이 생기고 속성이 채워지는지 확인.

## 범위 밖 / 후속

- **승인 카드 버튼 수명 관리** — 별도 PR. Slack `response_url`은 발급 후 30분만 유효한 반면 카드 TTL은 24시간이라, 30분이 지난 뒤 누르면 처리는 되지만 결과를 돌려보낼 통로가 없다. 만료된 카드의 버튼이 계속 눌리는 상태로 남는 문제도 같은 뿌리다. 해결하려면 카드가 게시된 채널·메시지 식별자를 저장해야 하므로 스키마 변경이 따른다.
- **자연어 경로의 속성 설정 실패** — 대상 DB에 `발행일`·`요약` 속성이 없어 `buildBlogPublishProperties`가 항상 거부당한다. 실행 4건 모두 `published: false`였다. DB에 속성을 추가할지, 없는 속성을 건너뛰게 할지 판단이 필요하다.
- **운영성 감사 G7(자연어 경로의 승인 게이트 부재)** — 게이트를 추가하지 않는 것으로 종결한다. 근거: ①`2026-06-17-notion-blog-autopublish-design.md:11`에 "완전 자동(PreviewGate 게이트 없음)"이 확정 결정으로 기록돼 있다 ②대상 DB가 웹에 공개돼 있지 않아 `상태=발행`에 외부 노출 효과가 없다 ③속성 부재로 그 코드가 성공한 적이 없다 ④경로가 사람의 명시적 요청으로만 시작되며 cron 트리거가 없다. 승인 게이트를 더하면 없는 위험을 막는 대가로 만료 실패 모드만 늘어난다.
