# 보유 종목 모니터링 설계 (Spec 1)

작성일: 2026-07-22

## 1. 배경과 목표

개인이 보유한 주식을 매일 자동으로 점검해, 이상이 있을 때만 Slack DM으로 알리는 기능을 만든다. 이대리 레포에는 금융 도메인 코드가 전혀 없으므로 신규 도메인이다.

최종적으로는 네 가지를 지향한다 — 보유 종목 모니터링, 신규 종목 스크리닝, 시장·뉴스 브리핑, 매매 타이밍 신호. 이 문서는 그중 **보유 종목 모니터링과 그 아래 데이터 계층만** 다룬다.

이 기능은 **혼자 쓰는 개인 도구**다. 외부 공개나 수익화 계획이 없으므로 투자자문업 등록이나 면책 고지는 범위에 없다. 다만 이 전제가 바뀌면 데이터 라이선스부터 다시 검토해야 한다.

### 성공 기준

3거래일 연속으로 지정 시각에 Slack DM이 도착하고, 이상이 없는 날은 한 줄 요약이, 발화한 날은 어떤 규칙이 어떤 값으로 걸렸는지가 함께 온다. 알림에 실린 종가와 평단 대비 손익이 증권사 앱 화면과 일치한다. 수집이 실패한 날에는 침묵하지 않고 실패 사실이 DM으로 온다.

마지막 조건이 특히 중요하다. "이상 없으면 조용하다"는 방식의 가장 위험한 실패 모드는 파이프라인이 죽어도 조용하다는 것이다. 정상적인 침묵과 고장난 침묵은 반드시 구분되어야 한다.

## 2. 선행 조건: Node 22 업그레이드

`yahoo-finance2`는 런타임에서 Node 22 이상을 요구한다. 현재 레포는 로컬과 CI 모두 Node 20이다(`.github/workflows/ci.yml:31`). Node 20에서도 실제 호출은 성공하지만 실행할 때마다 `Unsupported environment` 경고가 나오고, 라이브러리의 유지보수 계약 밖이라 언제 깨져도 이상하지 않다.

따라서 **레포 런타임을 Node 22로 올리는 작업을 별도 PR로 선행**한다. 이 작업은 이대리 전체에 영향을 주므로 주식 기능과 같은 PR에 섞지 않는다. 네이티브 모듈(Prisma, tree-sitter 등) 재빌드가 필요할 수 있다.

## 3. 데이터 소스

`yahoo-finance2`를 단일 어댑터로 사용한다. 국내와 미국 종목을 같은 인터페이스로 조회할 수 있고, 증권계좌나 API 키 발급 없이 즉시 시작할 수 있다.

실제 호출로 확인한 사실은 다음과 같다.

| 확인 항목 | 결과 |
|---|---|
| 국내 종목 | `005930.KS`(KOSPI), `247540.KQ`(KOSDAQ) 정상 조회 |
| 지수 | `^KS11` 조회 가능 |
| 환율 | `KRW=X` 조회 가능 (USD/KRW) |
| 과거 데이터 | 삼성전자 2005년부터 5,334봉 소급 조회 |
| 휴장일 | 거래일만 반환. 별도 휴장일 캘린더 불필요 |
| 배치 조회 | 5종목 1회 호출 869ms |
| 국내 시세 지연 | 20분 (미국은 실시간) |

이 선택에는 명확한 한계가 있다. 비공식 경로이므로 응답 형식이 바뀌면 깨지고, 가용성이나 요청 한도에 대한 보장이 없다. 라이브러리 릴리스 노트를 보면 Yahoo 응답 스키마 변경에 대응하는 수정이 월 단위로 발생한다. 개인 내부 사용으로 범위를 한정하되, 언제든 차단될 수 있음을 감수하는 선택으로 기록한다.

한국투자증권 API는 정확하고 실시간이지만 계좌 개설과 앱키 발급, 토큰 갱신 로직이 필요해 첫 조각에는 과중하다고 판단했다. 다만 20분 지연 때문에 **매매 타이밍 신호 단계에 가려면 전환이 사실상 강제**되므로, 포트 경계를 지켜 교체 가능한 상태로 둔다.

### 3.1 심볼 검증은 선택이 아니라 필수다

이 소스의 가장 위험한 특성이다. **잘못된 심볼이 에러를 내지 않고 엉뚱한 값을 반환한다.**

```
005930.KS → SamsungElec        273,500 KRW   정상
005930.KQ → 005930.KQ,0P00...   84,400 KRW   예외 없이 다른 값
005930    → undefined                        throw조차 하지 않음
005935.KS → SamsungElec(1P)    190,800 KRW   한 자리 차이로 우선주
```

6자리 코드만으로는 KOSPI와 KOSDAQ을 구분할 수 없다. 접미사를 잘못 붙이면 3배 틀린 원화 가격이 조용히 저장되고, 평단 대비 손익과 이동평균이 전부 엉뚱한 종목 기준으로 계산된다. 원화 단위의 그럴듯한 숫자라 사람이 알아채기 어렵다.

**종목 등록 시점에 반드시 검증한다.** 조회 결과가 `undefined`이거나 `shortName`이 심볼 문자열 또는 콤마 목록 형태이면 거부한다. 위 오염 사례는 이 두 조건으로 모두 걸러진다. 등록 전에는 조회된 종목명·거래소·통화를 사람이 눈으로 확인하는 절차를 둔다.

### 3.2 과거 가격은 소급해서 다시 쓰인다

Yahoo는 액면분할이 발생하면 과거 데이터를 통째로 재작성한다. 삼성전자의 2005년 종가가 9,020으로 조회되는데, 당시 실제 종가는 45만원대다. 2018년 50:1 분할이 소급 반영된 값이다.

따라서 오늘 저장한 종가는 미래에 분할이 일어나면 재조회 시 다른 값이 된다. 저장해 둔 과거 종가로 이동평균이나 전일 대비를 계산하면 **분할 다음 날 -50% 폭락 알림**이 나간다.

대응: `close`와 `adjClose`를 모두 저장하고 지표 계산은 `adjClose`를 기준으로 한다.

주기적 재동기화(최근 250봉을 다시 받아 upsert)는 **Spec 2로 미룬다.** 이번 범위는 판정에 최근 5봉만 쓰고 매 실행마다 새로 받으므로, 저장된 과거 값이 지표를 오염시킬 경로가 없다. 스키마에 `lastResyncedAt`을 미리 두어 나중에 재동기화를 붙일 때 마이그레이션이 필요 없게 한다.

## 4. 범위

### 이번에 만드는 것

국내 시장만. 보유 종목 등록(스크립트), 일별 시세 수집과 저장, 두 가지 판정 규칙, Slack DM 발송, 수집 실패 알림.

### 이번에 만들지 않는 것

미국 시장, 슬래시 커맨드, LLM 서술, 알림 사후 성과 채점, 이동평균·거래량·지수 대비 규칙, 신규 종목 스크리닝, 뉴스 브리핑.

각각의 이유는 다음과 같다.

**미국 시장**은 통화·환율·서머타임·별도 휴장일을 한꺼번에 끌고 온다. 국내만으로 파이프라인이 도는 것을 먼저 확인한 뒤 붙인다.

**슬래시 커맨드**는 이 레포에서 저렴하지 않다. AGENTS.md의 신규 에이전트 체크리스트가 `ResponseCode` enum, `TriggerType` enum, `AgentType`과 모델 라우팅 등록, `/retry-run` 분기, Slack manifest 등록, README 표까지 14항목을 요구한다. 보유 종목 서너 개를 넣기 위한 비용으로는 과하다. 종목 교체가 실제로 잦다는 것이 관측되면 그때 추가한다.

**LLM 서술**은 이 레포의 모델 라우터에 fallback이 없어(`FALLBACK_OF`가 비어 있고 `route()`가 즉시 throw) 서술 실패가 알림 자체를 죽인다. 숫자만 있는 짧은 보고에 10~40초 CLI 호출과 실패 지점을 추가할 이유가 없다. 나중에 붙일 때는 `HumanizeService`의 `noFallback: true` 패턴을 따라, 결정론 텍스트를 먼저 완성한 뒤 윤문을 try/catch로 감싸 실패 시 원문을 그대로 보낸다.

**사후 채점**은 지금 만들 필요가 없다. 알림 발생 시점의 값만 남겨 두면 성과는 나중에 과거 시세로 소급 계산할 수 있다. 보유 종목 수를 감안하면 통계적으로 의미 있는 표본이 모이는 데도 오랜 시간이 걸린다.

**규칙 3종**은 임계값을 지금 정할 근거가 없다. 실제 알림을 며칠 받아본 뒤 정한다.

### 축적에 대한 정정

초기 검토에서 "시계열을 오늘부터 쌓지 않으면 나중에도 못 만든다"고 판단했으나 이는 사실이 아니다. 일별 시세는 20년 이상 소급 조회된다.

실제로 되돌릴 수 없는 것은 따로 있다. **보유·매매 이력**, **알림 발화 이력**, 그리고 상장폐지된 종목의 과거 데이터다(Yahoo는 상장폐지 시 과거 이력까지 제거한다). 축적의 목적을 "가격 확보"가 아니라 "그 시점의 내 상태 보존"으로 정정한다. 따라서 Layer 0에서 우선순위가 높은 것은 수집 cron이 아니라 보유 이력 기록이다.

## 5. 구성

### 5.1 모듈

- `src/market-data/` — 외부 시세 조회. CODE_RULES §1-8의 "외부 API는 독립 도메인으로 취급" 규칙에 따라 분리한다.
  - `domain/port/market-data.port.ts` — `MARKET_DATA_PORT` Symbol과 인터페이스
  - `infrastructure/yahoo-finance.market-data.client.ts` — `YahooFinanceMarketDataClient`
- `src/agent/stock/`
  - `domain/stock-anomaly.ts` — 순수 판정 함수
  - `infrastructure/stock-monitor.formatter.ts` — Slack 문자열
  - `infrastructure/tasks/stock-monitor.autopilot-task.ts` — 수집·판정·전달 조립

판정과 출력을 한 단위로 묶지 않는다. `src/autopilot/domain/run-retro.anomaly.ts`가 순수 판정기를 도메인에 두고 포맷과 전달을 분리한 선례를 따른다.

### 5.2 포트

```ts
type ResolvedInstrument = {
  yahooSymbol: string;  // 005930.KS
  code: string;         // 005930
  market: MarketCode;   // KOSPI | KOSDAQ
  name: string;         // SamsungElec
  currency: string;     // KRW
};

type DailyBar = {
  tradeDate: Date; // 거래소 로컬 날짜
  close: Prisma.Decimal;
  adjClose: Prisma.Decimal;
  volume: bigint;
  currency: string;
  isFinal: boolean; // 장중 부분봉 여부
};

interface MarketDataPort {
  resolveSymbol(symbol: string): Promise<ResolvedInstrument | null>;
  fetchDailyBars(symbol: string, days: number): Promise<DailyBar[]>;
}
```

`resolveSymbol`은 3.1의 검증을 담당한다. 조회 결과가 없거나 `shortName`이 심볼 문자열 또는 콤마 목록이면 `null`을 반환한다. 종목 등록 스크립트는 `null`을 받으면 저장하지 않고 중단한다. 이 함수는 심볼 문자열을 그대로 받으며, 종목명 검색은 이번 범위가 아니다.

라이브러리 응답 객체를 그대로 흘려보내지 않는다. 어댑터 안에서 도메인 타입으로 변환하며 검증하고, 필드가 없거나 타입이 바뀌면 명시적 에러를 던진다. 이 변환 함수가 외부 API 형식 변경을 잡아내는 유일한 자동 방어선이므로, 저장해 둔 실제 응답 JSON으로 테스트한다.

라이브러리는 v3부터 인스턴스화가 필요하다(`new YahooFinance()`). 정적 호출은 실패한다. 어댑터에서 한 번 생성해 DI로 공유한다.

### 5.3 스키마

핵심 결정 세 가지.

**금액은 `Decimal @db.Decimal(18,4)`로 둔다.** 현재 `prisma/schema.prisma`에 Decimal 사용처가 없어 관성적으로 Float를 쓸 위험이 크다. 평단과 수익률에 부동소수점 오차가 누적되면 안 된다.

**티커를 기본키로 쓰지 않는다.** 미국 티커 변경(FB→META), 국내 이전상장, 합병·상장폐지로 심볼은 바뀐다. 내부 surrogate id를 기본키로 두고 `yahooSymbol`은 가변 필드로 둔다. 유니크는 `(market, code)`.

**중복 저장을 막는다.** `DailyPrice`에 `@@unique([tickerId, tradeDate])`를 걸고 upsert한다. CODE_RULES §12-5가 반복 실행 시 중복 저장 방지를 필수로 규정한다. 중복 유입 경로는 실재한다 — 스케줄러의 `attempts: 2` 재시도, 그리고 휴장일에 직전 거래일 봉을 다시 받는 경우다.

`tradeDate`는 cron 실행일이 아니라 **봉 타임스탬프를 거래소 타임존에서 정규화**해 산출하고 `@db.Date`로 둔다. `planDate DateTime @unique @db.Date`(`prisma/schema.prisma:112`)가 같은 패턴이다.

| 모델 | 주요 필드 |
|---|---|
| `Ticker` | id(surrogate), code, market, yahooSymbol, name, currency |
| `DailyPrice` | tickerId, tradeDate, close, adjClose, volume, isFinal, fetchedAt, lastResyncedAt |
| `Holding` | tickerId, quantity, avgPrice, currency, effectiveDate |
| `StockAlert` | tickerId, tradeDate, ruleId, ruleVersion, triggeredValue, threshold, firedAt, deliveryState |

`StockAlert`에 규칙 버전과 당시 입력값을 남기는 이유는, 임계값을 나중에 바꿔도 과거 알림이 왜 발생했는지 재현할 수 있어야 하기 때문이다.

### 5.4 판정 규칙

두 가지만 둔다. 임계값은 실제 분포를 측정해 정했다.

| 규칙 | 발화 조건 | 임계값 |
|---|---|---|
| `daily-change` | 전일 종가 대비 등락률 절대값이 기준 초과 | **±8%** |
| `avg-price-breach` | 평단 대비 손익률이 임계 구간에 **최초 진입**할 때 | **-20% / +30%** |

#### 전일 대비 임계값의 근거

최근 250거래일의 등락률 분포를 측정한 결과는 다음과 같다.

| 종목 | 중앙값 | 상위 10% | ±5% 발화 | ±8% 발화 | ±10% 발화 |
|---|---|---|---|---|---|
| 삼성전자 | 2.23% | 6.92% | 연 48회 | 연 18회 | 연 9회 |
| 에코프로비엠 | 2.18% | 7.68% | 연 61회 | 연 21회 | 연 10회 |
| SK하이닉스 | 2.81% | 8.83% | 연 76회 | 연 33회 | 연 17회 |

±5%는 주 1회 이상 울린다. "이상할 때만 알린다"는 목표와 맞지 않고, 알림 피로로 기능이 방치될 가능성이 높다. ±8%는 월 1.5~2.7회 수준으로, 실제로 확인할 가치가 있는 빈도다.

측정 과정에서 종목별 변동성 차이도 드러났다. 중앙값은 2.2~2.8%로 비슷하지만 상위 5% 구간은 9~11%로 벌어진다. 이상적으로는 종목별 변동성으로 정규화해야 하지만, 개인 도구 규모에서 그 복잡도를 감당할 이유가 없다. 고정 임계값으로 시작하고 특정 종목이 계속 소음을 내면 그 종목만 조정한다.

#### 평단 대비는 이벤트가 아니라 상태다

이 규칙은 단순 비교로 만들면 안 된다. 평단 대비 손익률은 한 번 -20%를 넘으면 회복할 때까지 계속 -20% 아래에 머문다. 매일 판정하면 **같은 사실을 매일 반복 발송**하게 된다.

따라서 **임계 구간에 최초 진입하는 순간에만** 발화한다. 어제는 임계 밖이었는데 오늘 안으로 들어온 경우다. 반대로 구간을 벗어났다가 다시 진입하면 새로운 사건으로 취급한다.

```ts
detectAvgPriceBreach(
  today: DailyBar,
  yesterday: DailyBar,
  holding: Holding,
  thresholds: { lower: number; upper: number },
): StockAnomaly | null
```

어제 상태는 별도로 저장하지 않는다. 전일 종가와 평단으로 다시 계산하면 되므로 상태 필드를 두면 중복 소스가 된다.

매수 후 1년 최대낙폭의 중앙값이 삼성전자 -5.9%, 에코프로비엠 -38%로 종목 성격에 따라 크게 갈린다. -20%는 대형주에서는 드물게, 고변동주에서는 의미 있는 하락에서 걸리는 지점이다.

#### 공통 사항

각 규칙은 순수 함수로 구현하고 필요한 시각은 인자로 받는다. 함수 내부에서 현재 시각을 읽으면 테스트할 수 없다. 규칙마다 발화·미발화·경계값 세 가지 테스트를 먼저 작성한다.

`daily-change`는 상태가 아니라 그날의 사건이므로 중복 억제가 필요 없다. 다만 같은 종목이 여러 날 연속으로 걸리면 알림 피로가 생기는지 2주간 관찰하고, 필요하면 연속 발화에 쿨다운을 넣는다.

### 5.5 스케줄

autopilot 플레이북 항목으로 등록한다. 별도 큐로 만들면 플레이북 무결성 검증과 운영 감시 밖에 놓이고, 미등록 cron 알람에 걸린다.

- 실행 시각: `Asia/Seoul` 기준 **17:10**
- `digestGroup`은 지정하지 않는다. 같은 그룹은 스케줄과 타임존 일치가 강제된다.

16:30이 아니라 17:10인 이유는 두 가지다. 국내 시세가 20분 지연되고, **수능일에는 정규장이 10:00~16:30으로 순연**된다(2025-11-13 실제 적용). 16:30에 수집하면 그날은 장중 부분봉을 종가로 저장하게 된다.

추가 가드로 조회 결과의 `marketState`가 `CLOSED`가 아니면 저장을 건너뛴다. 장중 부분봉은 거래량이 정규 일봉의 일부에 불과해 거래량 기반 판정을 오염시킨다.

### 5.6 휴장일

휴장일 테이블을 만들지 않는다. 임시공휴일은 단기에 공지되어 하드코딩이 불가능하고, 개인 도구 규모에서 캘린더를 유지보수할 이유가 없다.

**데이터로 판정한다.** 새로 수집한 마지막 봉의 거래일이 직전 수집분과 같으면 휴장 또는 미갱신으로 분류하고 판정을 건너뛴다. 한 줄 요약에는 `휴장(추정)`으로 표기한다.

이 처리가 없으면 휴장일에 전일 대비가 0%로 계산되어 "이상 없음" 알림이 나가고, 사용자는 시스템이 정상 동작한 것으로 오인한다.

### 5.7 출력과 실패 처리

결정론 포맷터로만 문자열을 만든다.

- 이상 없음: 한 줄 요약 (점검한 종목 수, 마지막 거래일, 총 평가손익)
- 발화: 종목별로 어떤 규칙이 어떤 값으로 걸렸는지, 임계값과 함께
- 휴장 추정: 한 줄, 판정 생략 명시
- **수집 실패: 실패한 종목과 사유를 반드시 발송**

수집이 부분 실패했을 때 성공한 종목만으로 정상 알림을 보내지 않는다. 실패한 종목과 사유를 같은 메시지에 함께 싣는다.

데이터 품질 검사는 어댑터의 변환 단계에서 한다. 거래일과 종가가 없는 봉은 도메인 타입으로 변환되지 않고 걸러진다. OHLC 정합성 검사(`high >= max(open, close)` 등)는 이번 범위에서 하지 않는다 — 판정에 종가와 조정종가만 쓰므로 시가·고가·저가를 저장하지 않기 때문이다. 이 필드들이 필요해지는 시점(이동평균, 거래량 규칙)에 함께 도입한다.

## 6. 검증

구현 완료의 기준은 1절의 성공 기준이며, 다음 게이트를 통과해야 한다.

- `pnpm lint:check && pnpm test && pnpm build` 세 개 모두 통과
- 새 env를 추가하면 `.env.example`, `.env`, `src/config/app.config.ts`, README 네 곳을 동기화하고 `pnpm check:env` 통과
- `AgentType`이나 에이전트 레지스트리를 건드리면 `pnpm docs:sync` 후 `pnpm docs:check` 통과 — 이 두 가지는 로컬 3중 검증에 포함되지 않고 CI에서만 잡힌다

시세 조회 스모크 테스트는 실제 네트워크를 타므로 `pnpm test`에서 분리한다. 이 레포의 `test` 스크립트는 jest를 두 번 실행하는 구조라 경로 필터가 동작하지 않는다. 단일 파일 실행은 `pnpm exec jest src/...`를 쓴다.

기능은 `STOCK_MONITOR_ENABLED` 플래그로 감싸고 기본값을 false로 둔다. 수집이 연속 실패하면 자동으로 비활성화하고 경보한다.

## 8. 확인하지 못한 것

- KRX 시간외단일가(16:00~18:00)가 일별 시세에 반영되는지 — 측정 시각이 장중이라 확인하지 못했다. 17:10 스케줄을 적용할 때 함께 실측한다.
- 휴장일에 조회가 빈 배열을 주는지 직전 봉을 반복하는지 — 다음 휴장일에 확인한다. 5.6의 판정 방식은 두 경우 모두에서 동작하지만 표기 문구가 달라진다.
- 요청 한도의 실제 임계 — 보유 종목 수가 적어 이번 범위에서는 문제되지 않을 가능성이 높다.
- DART 인증키는 개인 신청 시 즉시 발급되므로 선행 준비 항목이 아니다. 재무 데이터를 쓰는 단계에서 신청한다.
